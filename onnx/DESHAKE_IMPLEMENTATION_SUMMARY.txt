================================================================================
PIPELINE POSITIONING
================================================================================

Deshake is a POST-PROCESS STAGE that operates AFTER YUV/RGB conversion.

Pipeline Order:
1. Demosaic
2. Black Level Correction
3. White Balance
4. CCM (Color Correction Matrix)
5. Gamma Correction
6. YUV/RGB Conversion
7. Deshake (Video Stabilization) ← POST-PROCESS STAGE
8. Output

Why Post-Process?
- Deshake operates on final color-corrected frames
- Motion estimation works better on YUV/RGB than raw bayer
- Stabilization should be applied after all color corrections
- Preserves color accuracy during motion compensation

================================================================================
ALGORITHMS CREATED (PHASE 1)
================================================================================

1. DeshakeBase (family: deshake_base)
   - Global motion compensation
   - Single motion vector for entire frame
   - Complexity: ~15-20 ONNX nodes
   - Use Case: Real-time video stabilization on mobile devices

2. DeshakeV1 (family: deshake_v1)
   - Grid-based motion compensation
   - Per-grid motion vectors with smoothing
   - Complexity: ~30-40 ONNX nodes
   - Use Case: High-quality video stabilization

3. DeshakeV2 (family: deshake_v2)
   - Feature-based motion compensation with RANSAC
   - Homography estimation with outlier rejection
   - Complexity: ~50-60 ONNX nodes
   - Use Case: Professional video stabilization

ALGORITHM DETAILS
================================================================================

DeshakeBase
------------
Purpose: Global motion compensation for video stabilization

Pipeline Position: After YUV/RGB conversion, before output

Inputs:
- current_frame [n,3,h,w] : Current video frame (YUV or RGB)
- prev_frame [n,3,h,w] : Previous video frame (YUV or RGB)
- motion_strength [1] : Motion compensation strength (0-1)

Outputs:
- stabilized_frame [n,3,h,w] : Stabilized frame (YUV or RGB)
- motion_vector [2] : Global motion vector (dx, dy)

Algorithm:
1. Convert frames to grayscale (for motion estimation)
2. Calculate block matching between frames
3. Compute global motion vector (dx, dy)
4. Apply motion strength scaling
5. Create sampling grid from motion vector
6. Apply GridSample for motion compensation
7. Handle borders (padding or crop)
8. Output stabilized frame

Motion Model:
  dx = diff_mean * motion_strength
  dy = diff_mean * motion_strength
  motion_vector = [dx, dy]

Use Case: Real-time video stabilization on mobile devices

Complexity: ~15-20 ONNX nodes

DeshakeV1
--------
Purpose: Grid-based motion compensation for video stabilization

Pipeline Position: After YUV/RGB conversion, before output

Inputs:
- current_frame [n,3,h,w] : Current video frame (YUV or RGB)
- prev_frame [n,3,h,w] : Previous video frame (YUV or RGB)
- motion_strength [1] : Motion compensation strength (0-1)
- smoothing_factor [1] : Temporal smoothing factor (0-1)
- grid_size [1] : Grid size (e.g., 8)

Outputs:
- stabilized_frame [n,3,h,w] : Stabilized frame (YUV or RGB)
- motion_grid [grid_h,grid_w,2] : Per-grid motion vectors

Algorithm:
1. Convert frames to grayscale (for motion estimation)
2. Calculate difference between frames
3. Resize to grid size
4. Calculate per-grid motion vectors
5. Apply spatial smoothing across grid
6. Apply temporal smoothing
7. Apply motion strength scaling
8. Resize motion grid to full resolution
9. Create sampling grid from motion grid
10. Apply GridSample for motion compensation
11. Handle borders
12. Output stabilized frame

Motion Model:
  motion_grid[i,j] = smooth(diff[i,j], spatial_alpha, temporal_alpha)
  motion_grid = motion_grid * motion_strength

Use Case: High-quality video stabilization

Complexity: ~30-40 ONNX nodes

DeshakeV2
--------
Purpose: Feature-based motion compensation with RANSAC

Pipeline Position: After YUV/RGB conversion, before output

Inputs:
- current_frame [n,3,h,w] : Current video frame (YUV or RGB)
- prev_frame [n,3,h,w] : Previous video frame (YUV or RGB)
- motion_strength [1] : Motion compensation strength (0-1)
- smoothing_factor [1] : Temporal smoothing factor (0-1)
- ransac_threshold [1] : RANSAC outlier threshold

Outputs:
- stabilized_frame [n,3,h,w] : Stabilized frame (YUV or RGB)
- homography [3,3] : 3x3 homography matrix
- inlier_count [1] : Number of inlier features

Algorithm:
1. Convert frames to grayscale (for motion estimation)
2. Extract features (e.g., corners, keypoints)
3. Match features between frames
4. Estimate homography using RANSAC
5. Filter outliers based on threshold
6. Apply temporal smoothing to homography
7. Create sampling grid from homography
8. Apply GridSample for motion compensation
9. Handle borders
10. Output stabilized frame

Motion Model:
  H = estimate_homography(features, ransac_threshold)
  H_smoothed = smooth(H, temporal_alpha)
  stabilized_frame = warp(current_frame, H_smoothed)

Use Case: Professional video stabilization

Complexity: ~50-60 ONNX nodes

================================================================================
PARAMETERS
================================================================================

Common Parameters:
------------------
- motion_strength [1] : Motion compensation strength (0-1)
  - 0.0: No compensation
  - 0.5: Moderate compensation
  - 1.0: Full compensation

- smoothing_factor [1] : Temporal smoothing factor (0-1)
  - 0.0: No smoothing (responsive)
  - 0.5: Moderate smoothing
  - 1.0: Maximum smoothing (stable)

DeshakeV1 Specific:
-------------------
- grid_size [1] : Grid size for motion estimation
  - 4: 4x4 grid (coarse, fast)
  - 8: 8x8 grid (balanced)
  - 16: 16x16 grid (fine, slow)

DeshakeV2 Specific:
-------------------
- ransac_threshold [1] : RANSAC outlier threshold
  - 0.1: Strict (few inliers)
  - 1.0: Moderate
  - 10.0: Loose (many inliers)

================================================================================
GRIDSAMPLE OPERATOR
================================================================================

The GridSample operator is used for motion compensation:

Input:
- input: [n, c, h, w] - Input image tensor (YUV or RGB)
- grid: [n, h_out, w_out, 2] - Sampling grid

Grid Format:
- grid[..., 0] = x coordinates (normalized to [-1, 1])
- grid[..., 1] = y coordinates (normalized to [-1, 1])

Parameters:
- mode: bilinear (interpolation mode)
- padding_mode: zeros (out-of-bounds handling)
- align_corners: 1 (coordinate system alignment)

Output:
- output: [n, c, h_out, w_out] - Stabilized image (YUV or RGB)

Motion Compensation:
  x_sample = x - dx  (inverse motion)
  y_sample = y - dy
  grid = [x_sample, y_sample]
  stabilized = GridSample(input, grid)

================================================================================
BORDER HANDLING
================================================================================

When applying motion compensation, regions outside the original frame
need to be handled. Common strategies:

1. **Padding**: Fill borders with black or mirrored pixels
   - Simple to implement
   - May introduce artifacts

2. **Cropping**: Crop to valid region (reduces resolution)
   - Clean output
   - Reduces field of view

3. **Inpainting**: Fill borders with interpolated content
   - High quality
   - Computationally expensive

4. **Zoom**: Slight zoom to avoid borders
   - Balanced approach
   - Slight resolution loss

Current Implementation:
- Uses padding_mode='zeros' (black borders)
- Can be extended to support other strategies
